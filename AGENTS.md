# AGENTS.md

このファイルは、リポジトリ全体で作業するAIエージェント向けの開発ガイドです。ユーザーから明示された指示がある場合は、その指示を最優先してください。サブディレクトリにより具体的な `AGENTS.md` がある場合は、そちらも適用します。

## プロジェクト概要

FocusBoardは、勉強中にiPadのホーム画面から使うことを主目的とした、時計・ポモドーロ機能付きのReact PWAです。

- React 19、TypeScript、Vite、Vitestを使用する
- サーバー、アカウント、クラウド同期を必要としない
- 設定とタイマー状態はlocalStorage、アップロード背景はIndexedDBへ端末内保存する
- オフライン起動とGitHub Pagesのサブディレクトリ配信に対応する
- UIの主言語は日本語とする

## 重要な設計上の不変条件

- タイマーの進行はインターバル回数ではなく `endAt` と `Date.now()` の差から求める。画面非表示、スリープ、再読み込み後にも時刻ずれを増やさないこと。
- 保存済みデータは信用せず、`src/utils/storage.ts` で検証、範囲制限、既定値へのフォールバックを行うこと。
- `AppSettings.version`、`uiRevision`、`TimerState.version` を変更する場合は、既存ユーザーの保存データを維持する移行処理とテストを追加すること。
- localStorage、IndexedDB、AudioContext、Service WorkerなどのブラウザAPIが利用不可または失敗しても、アプリ本体が起動できる状態を保つこと。
- ユーザー画像は端末内だけで扱う。外部送信や外部サービス連携を追加しないこと。
- Viteの `base` とPWAのナビゲーションフォールバックを壊さず、ルート配信とGitHub Project Pagesの両方で動作させること。

## ディレクトリと責務

- `src/components/`: 表示とユーザー操作を担うReactコンポーネント
- `src/hooks/`: 時計、タイマー、設定、背景画像などの状態と副作用
- `src/types/`: 設定値やタイマー状態の型、既定値、選択肢
- `src/utils/`: 保存、時刻計算、背景色解析、PWA削除などの純粋処理または低レベル処理
- `public/`: マニフェスト、アイコン、組み込み背景
- `screenshots/`: READMEや動作確認用の画面資料
- `.github/workflows/`: GitHub Pagesのテスト、ビルド、デプロイ

責務をまたぐ大きな処理を `App.tsx` に集めず、表示はcomponent、再利用する状態管理はhook、純粋な変換や永続化はutilsへ配置してください。

## 実装規約

- TypeScriptのstrict設定を維持し、`any`、不要な型アサーション、型チェックの無効化を避ける。
- 既存コードに合わせ、2スペースインデント、ダブルクォート、セミコロンを使用する。
- コンポーネントは関数形式とし、既存のnamed exportを基本にする。`App` のdefault exportは既存仕様として維持する。
- stateは直接変更せず、更新前のstateから新しい値を返す。
- effectには必要な依存値とcleanupを用意する。タイマー、イベントリスナー、Object URLなどを残留させない。
- 意味のある名前を優先し、自明な処理を説明するコメントは追加しない。理由や制約がコードから読み取れない場合だけコメントを付ける。
- 新しい依存パッケージは、標準APIや既存依存で実現できない場合に限る。追加時は `package.json` と `package-lock.json` を一致させる。
- ユーザー向け機能、操作手順、保存形式、開発コマンドが変わる場合はREADMEも更新する。

## UI・アクセシビリティ

- iPadの縦向き・横向き、タッチ操作、Apple Pencil、safe areaを主要な対象とする。hoverだけに依存した操作を作らない。
- PC/iphone など複数の画面サイズでの動作を確認する。小さい画面では主要操作が隠れないことを優先する。
- 既存のパステル調、背景画像主体の画面、可変の文字色とアクセント色を尊重し、背景上で十分に読み取れる表示を保つ。
- 既存のレスポンシブ動作を維持し、小さい画面や表示倍率を上げた状態でも主要操作を隠さない。
- 操作要素には適切なHTML要素、フォーカス表示、キーボード操作、アクセシブルネームを用意する。
- ダイアログ、リストボックス、ラジオグループ、ライブ通知などは、既存のARIAとフォーカス復帰の動作を壊さない。
- 色だけで状態を伝えない。アニメーションを追加する場合は `prefers-reduced-motion` を考慮する。
- UIテキストを変更したときは、ロールやラベルを使うテストも更新する。

## テスト方針

- 不具合修正では、可能なら先に再現テストまたは回帰テストを追加する。
- pure utilityは境界値、不正データ、フォールバックを単体テストする。
- hooksは状態遷移、時刻経過、永続化、再読み込み相当の動作をテストする。
- componentsは実装詳細ではなく、ユーザーが認識できるrole、label、textと操作結果をテストする。
- localStorageなどの共有状態はテスト間で初期化し、fake timerやmockは必ず後始末する。
- スナップショットだけに頼らず、重要な操作と表示結果を明示的に検証する。

変更後は、影響範囲に応じて次を実行します。

```bash
npm test
npm run build
```

小さな変更では関連テストを先に実行してよいですが、完了前には原則として全テストとTypeScriptを含むproduction buildを確認してください。失敗した確認項目は隠さず、原因と未確認範囲を報告してください。

## 作業手順と安全性

- 作業開始時に `README.md`、`package.json`、関連コード、`git status --short` を確認する。
- ファイル探索とテキスト検索は、利用可能なら `rg --files` と `rg` を優先する。
- 変更は依頼範囲に絞る。無関係なリファクタリングや一括整形を混ぜない。
- ワークツリーにあるユーザーの変更を保持し、無関係な差分を上書き、削除、巻き戻ししない。
- 手編集には差分が明確になる方法を使い、生成物や一括機械変換を除いてパッチ形式の編集を優先する。
- `git reset --hard`、無断のファイル削除、強制pushなどの破壊的操作を行わない。
- 秘密情報、個人データ、端末内画像をログ、テストfixture、コミット対象へ含めない。
- セキュリティ上の懸念がある場合は、ユーザーに確認してから作業を進める。
- 作成するコードは、セキュリティ、プライバシー、アクセシビリティ、パフォーマンス、可読性、保守性を考慮する。
- 安全性を最優先事項とし、ユーザーの端末やデータを壊す可能性がある場合は、作業を中断して報告する。
- 診断やレビューだけを依頼された場合は、明示的な許可なく実装修正まで進めない。
- 実装を依頼された場合は、安全で依頼範囲内の修正、テスト、ビルドまで自律的に完了させる。

## エージェントからの報告

- Codexは `config.toml` の `model_verbosity = "low"` を前提とし、必要十分な情報だけを簡潔に伝える。
- ユーザーへの説明は日本語を基本とし、最初に結果を簡潔に伝える。
- 長い作業では、調査内容、前提、進捗を短く共有する。
- 完了報告には変更した内容、確認したテスト／ビルド、残る制約があればその内容を含める。
- 実行していない確認を実行済みとして扱わない。
